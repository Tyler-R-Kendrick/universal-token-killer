```python

runner = KompressRunner()
result = runner.compress("Your long text here...")

print(result.compression_ratio)
```

## Core Positioning

Kompress-base is an open-source local text-compression model for LLM context
windows. It is positioned as a drop-in replacement for LLMLingua-2 with better
self-reported quality and speed. It is not a full agent hook, proxy, serializer,
or artifact-recovery system by itself.

This differs from UTK's intended center:

- Kompress-base classifies natural-language tokens as keep/discard.
- Headroom wraps Kompress-base into a broad proxy/library compression stack.
- UTK mediates GitHub Copilot tool calls, persists raw artifacts, infers
  schemas, routes outputs, serializes compact responses, and exposes hook-first
  Copilot integration.

The overlap worth studying is local ML-based text compression. Kompress-base is
the strongest local competitor to UTK's current LLMLingua2 detok path for
natural-language fields, but its own model card says structured data, code, and
logs should use specialized compressors.

## Capability Inventory

| Capability | What it does | How Kompress-base implements it | UTK relevance |
|---|---|---|---|
| Local natural-language compression | Removes low-value words from text without a hosted API. | ModernBERT token classifier predicts keep/discard labels and reconstructs extractive compressed text. | Strong candidate for optional local detok provider, especially for long prose fields. |
| Content-adaptive compression | Lets compression ratio vary by text density. | Model decides what to retain rather than using a fixed target ratio. | Useful for natural-language summaries, but UTK still needs deterministic thresholds for tests and config. |
| Dual-head architecture | Combines token-level and span-level importance. | Token head is a binary classifier; span head is a CNN that boosts borderline tokens in important regions. | Interesting path for future local compression, but separate from UTK's schema router. |
| ModernBERT backbone | Handles longer input than LLMLingua-2's older mBERT baseline. | Based on `answerdotai/ModernBERT-base` with 8192-token context. | Good fit for before-reading-output-file text reduction after schema/template parsing. |
| Extractive labels | Ensures compressed output is a subset of original words in order. | Training labels are generated by asking a judge model to select words like highlighting, then greedily aligned back to original text. | Safer than abstractive summarization. UTK should prefer extractive compression for rewrite-sensitive fields. |
| Span preservation signal | Preserves important regions, not only isolated tokens. | Span head uses Conv1d layers over hidden states and a sigmoid output. | Similar goal to UTK protected-field policy, but learned rather than rule-enforced. |
| Headroom integration | Runs as default text compressor inside a broader context optimizer. | Headroom auto-downloads and runs the model on proxy/library requests. | Competitive context. UTK should stay narrower and hook-first while using local text compression where it helps. |
| ONNX artifact | Provides a smaller inference artifact. | Repo includes an LFS pointer for `onnx/kompress-int8.onnx` around 156MB. | Could inspire a faster local provider, but current UTK should avoid adding large model downloads by default. |
| Apache 2.0 license | Allows permissive use. | Hugging Face model card declares Apache 2.0. | Easier to consider as an optional provider than proprietary hosted APIs. |
| Self-reported evals | Compares quality and latency to LLMLingua-2. | Model card reports Claude-judged quality and latency against LLMLingua-2 and kompress-small. | Useful signal, but UTK needs repo-local deterministic evals before adopting. |

## Implementation Mechanics

### Model Repository

The model repo contains:

- `README.md`: model card, usage, training notes, limitations, and metrics;
- `config.json`: ModernBERT config plus `HeadroomCompressor` architecture
  marker;
- `tokenizer.json` and `tokenizer_config.json`: tokenizer with
  `model_max_length` of 8192;
- `model.safetensors`: Git LFS pointer, 600,014,148 bytes;
- `onnx/kompress-int8.onnx`: Git LFS pointer, 155,950,298 bytes;
- `training_args.bin`: Git LFS pointer.

The checked config names:

- `architectures`: `HeadroomCompressor`;
- `model_type`: `modernbert`;
- `hidden_size`: 768;
- `num_hidden_layers`: 22;
- `num_attention_heads`: 12;
- `max_position_embeddings`: 8192;
- `num_labels`: 2;
- `task_specific_params.token_compression.span_cnn_hidden`: 256;
- `task_specific_params.token_compression.span_kernel_sizes`: `[5, 3]`;
- `task_specific_params.token_compression.span_loss_weight`: 0.3.

### Architecture

The model card describes Kompress-base as a dual-head ModernBERT model:

- base: `answerdotai/ModernBERT-base`;
- token head: `Linear(768, 2)` for binary keep/discard classification;
- span head: `Conv1d(768->256, k=5) -> GELU -> Conv1d(256->1, k=3) -> Sigmoid`;
- total parameters: about 150M.

This means Kompress is extractive, not abstractive. It does not rewrite facts
into new prose. It selects original tokens/words and drops the rest. That is a
better safety profile than prompt-only "make this shorter" compression, but it
still can remove a token that later matters.

### Training And Labels

The model card reports 215K extractive compression labels from eight datasets:

- LMSYS-Chat-1M;
- CNN/DailyMail;
- WikiHow;
- MeetingBank;
- XSum;
- GovReport;
- ArXiv;
- SAMSum.

Labels were generated by Claude Sonnet 4.6. The key training lesson is that
labels must be strictly extractive. Earlier rephrasing-style labels failed
because the compressed text could not align back to original words, producing
bad keep ratios. The documented fix was to ask the labeler to select words as
if highlighting them.

This is directly relevant to UTK's detok path: local text rewriting should be
extractive whenever the downstream model may need exact recovery or when raw
artifacts are available separately.

### Evaluation Claims

The model card reports:

- `kompress-base`: 7.9/10 quality, 84ms median latency on Apple Silicon MPS,
  600MB, 150M params;
- `kompress-small`: 7.4/10 quality, 13-29ms ONNX latency, 279MB, 70M params;
- LLMLingua-2: 5.9/10 quality, 117ms latency, 710MB, 179M params;
- Claude Code sessions: 7.3/10 for kompress-base, 7.4/10 for kompress-small,
  6.2/10 for LLMLingua-2.

These are self-reported, Claude-judged scores. They are useful for prioritizing
experiments but should not be treated as proof that Kompress preserves UTK's
required facts, schemas, paths, commands, or tool-output semantics.

### Limitations

The model card explicitly says:

- English only;
- best on natural-language text;
- structured data such as JSON, code, and logs should use specialized
  compressors;
- compression ratio varies by content, roughly 60-80 percent kept for dense
  text and 40-60 percent kept for verbose text.

This aligns with UTK's architecture: schema route first, serializer second,
local language compressor only after structured parsing and protected-field
policy have done their job.

## Competitive Opportunities For UTK

1. Add Kompress-base as an optional local detok provider, not the default core
   serializer. It fits natural-language fields, not structured tool outputs.
2. Build an eval adapter that compares LLMLingua2 vs Kompress-base on UTK's
   actual Copilot prose fields, tool descriptions, long notes, and session-skill
   references.
3. Keep schema-aware serializers ahead of ML compressors. TOON and compressed
   JSON should handle objects, arrays, errors, diffs, and command outputs before
   text compression is considered.
4. Use extractive compression as a safety requirement for local rewrite
   providers where possible. Dropped content must remain recoverable from raw
   artifacts.
5. Consider ONNX only behind an explicit install/config path. A 156MB ONNX file
   is too heavy for transparent default setup, but reasonable for opt-in local
   acceleration.
6. Treat content-adaptive compression as a runtime metric, not an unchecked
   behavior. UTK should record observed keep ratio, tokens saved, and required
   fact retention per mediated call.
7. Add specialized structured compressors for the areas Kompress excludes:
   JSON, code, logs, diffs, shell outputs, and non-shell tool objects.
8. Compete on recovery. Kompress drops tokens; UTK should pair any drop with
   `.utk/` raw artifact ids and route metadata.
9. Compare against self-reported Claude-judged quality with deterministic local
   checks. UTK's CI should fail on missing facts even if an LLM judge likes the
   prose.
10. Keep UTK non-proxy. Headroom packages Kompress through a broad proxy stack;
   UTK should expose it, if at all, through hook-local provider configuration.

## Risks To Avoid

- Do not use Kompress-base on JSON, code, logs, command output, diffs, patches,
  paths, ids, schemas, or tool arguments without a specialized protected policy.
- Do not silently download hundreds of MB of model weights during `utk-init`.
- Do not replace deterministic serializer tests with Claude-judged quality
  scores.
- Do not assume content-adaptive compression meets a requested token budget.
- Do not lose the distinction between "compressed text still understandable"
  and "all required operational facts survived."
- Do not make UTK depend on Headroom or a proxy path to use local compression.
